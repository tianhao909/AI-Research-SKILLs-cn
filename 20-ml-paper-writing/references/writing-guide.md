## 叙事原则 | The Narrative Principle

### 来自 Neel Nanda

「论文是一篇简短、严谨、以证据为基础的技术故事，传达读者关心的核心结论。」

叙事建立在三个支柱之上，在引言结束时必须清晰明了：

**「是什么」**：一到三个符合统一主题的具体新颖主张。像「我们研究 X」这样模糊的贡献立即失败——审稿人需要精确的、可证伪的主张。

**「为什么」**：令人信服地支持这些主张的严格实证证据，包括诚实调优的强大基线，以及区分竞争假设的实验，而不仅仅是展示「还不错的结果」。

**「那又怎样」**：为什么读者应该关心，将你的贡献与社区公认重要的问题联系起来。

### 来自 Andrej Karpathy

「论文不是对你报告的实验的随机集合。论文推销的是之前不存在或不明显的一样东西。整个论文都以这个核心贡献为中心，精确组织。」

无论你展示的是新架构、理论结果还是对现有方法的改进理解，这都是适用的——NeurIPS 明确指出「原创性不一定需要全新方法」。

**实际意义**：如果你不能用一句话表达你的贡献，你的论文还没有成形。其他一切——实验、相关工作、讨论——都只服务于那个核心主张。

## 时间分配 | Time Allocation

### 来自 Neel Nanda

在大约**相同的时间**分配在以下各项：
1. 摘要
2. 引言
3. 图表
4. 其他所有内容

这不是夸张——大多数审稿人在到达你的方法部分之前就形成了初步判断。读者以可预测的顺序阅读你的论文：**标题 → 摘要 → 引言 → 图表 → 可能的其他部分。**

### 审稿人阅读模式

审稿人行为研究表明：
- 摘要 100% 被阅读
- 90%+ 的审稿人会浏览引言
- 大多数审稿人在方法之前查看图表
- 只有在建立兴趣后才会阅读完整方法

**含义**：将论文的价值前置。不要埋没贡献。

## 摘要写作公式 | Abstract Writing Formula

### Sebastian Farquhar 的 5 句公式

1. **你取得的成果**：「我们提出……」「我们证明……」「我们展示……」
2. **为什么这既困难又重要**
3. **你是如何做到的**（使用专业关键词以便发现）
4. **你有什么证据**
5. **你最突出的数字/结果**

### 示例（良好的摘要）

```
我们证明了过参数化神经网络的梯度下降以线性速率收敛到全局最优。[是什么]
这解决了一个关于深度学习为何有效的基础问题，尽管优化景观是非凸的。[为什么困难/重要]
我们的证明依赖于表明神经切核在训练过程中保持近似恒定，将问题简化为核回归。[如何做，包含关键词]
我们在 CIFAR-10 和 ImageNet 上验证了我们的理论，表明预测的收敛率与实验结果在 5% 以内匹配。[证据]
这是首次为具有实际深度和宽度的网络提供多项式时间收敛保证。[突出的结果]
```

### 应避免的内容

来自 Zachary Lipton：「如果第一句话可以添加到任何机器学习论文前面，就删除它。」

**删除这些开头**：
- 「大型语言模型取得了显著成功……」
- 「深度学习已经革新了……」
- 「近年来，神经网络……」

**改为从你的具体贡献开始。**

## 引言结构 | Introduction Structure

### 要求

- **最多 1-1.5 页**（双栏格式）
- **方法部分应在第 2-3 页开始**
- 必须包含 **2-4 点贡献列表**（每点最多 1-2 行）

### 结构模板

```markdown
1. 开场钩子（2-3 句话）
   - 阐述论文解决的问题
   - 为什么它现在很重要

2. 背景/挑战（1 段）
   - 是什么让这个问题困难？
   - 其他人尝试过什么？为什么还不够？

3. 你的方法（1 段）
   - 你做了什么不同的事？
   - 促成你贡献的关键洞察

4. 贡献要点（2-4 项）
   - 要具体且可证伪
   - 每点最多 1-2 行

5. 结果预览（2-3 句话）
   - 最令人印象深刻的数字
   - 评估范围

6. 论文组织（可选，1-2 句话）
   - 「第 2 节介绍……第 3 节描述……」
```

### 贡献要点：好的 vs 差的

**好的：**
- 我们证明了 X 在假设 Y 下以 O(n log n) 时间收敛
- 我们引入了 Z，一种将内存减少 40% 的 3 层架构
- 我们展示了 A 在基准 C 上比 B 优 15%

**差的：**
- 我们研究了 X 问题（不是贡献）
- 我们进行了大量实验（太模糊）
- 我们对该领域做出了多项贡献（什么都没说）

## 句子级别清晰度 | Sentence-Level Clarity

### 来自 Gopen & Swan：《科学写作的科学》

George Gopen 和 Judith Swan 1990 年的开创性论文指出，**读者对信息在文中出现的位置有结构性期望**。违反这些期望会使读者把精力花在结构上而不是内容上。

> 「如果作者想让读者理解其意思，作者必须了解读者需要什么。」

#### 读者期望的 7 项原则

**原则 1：主语动词临近原则**

将语法主语和动词放在一起。任何插入的内容都会被读作次要干扰。

**弱**：「该模型在 1 亿标记上训练并使用秩为 16 的 LoRA 在领域特定数据上进行微调，取得了最先进的结果」

**强**：「该模型在 1 亿标记上训练并使用 LoRA（秩 16）微调后，取得了最先进的结果」

**原则 2：强调位置（把最好的留到最后）**

读者自然强调**句子的最后部分**。把最重要的信息放在那里。

**弱**：「使用注意力时，准确率提升了 15%」
**强**：「使用注意力时，准确率提升了 **15%**」

**原则 3：主题位置（先说先说）**

句子的开头建立视角。把「主角」放在最前面——读者期望句子是关于最先出现的那个。

**弱**：「引入了一种计算对齐分数的新型注意力机制」
**强**：「为了解决对齐问题，我们引入了一种新型注意力机制」

**原则 4：旧信息在前，新信息在后**

将熟悉的信息（旧信息）放在主题位置以实现向后连接；将新信息放在强调位置以示强调。

**弱**：「Child 等人引入了稀疏注意力。标准注意力的二次复杂度促使了这项工作。」
**强**：「标准注意力具有二次复杂度。为了解决这个问题，Child 等人引入了稀疏注意力。」

**原则 5：每个单元一个功能**

每个话语单元（句子、段落、章节）应该只服务一个功能。如果有两点，用两个单元。

**原则 6：用动词表达动作**

用动词而不是名词化形式来表达每个句子的动作。

**弱**：「我们对结果进行了分析」（名词化）
**强**：「我们分析了结果」（动作在动词中）

**原则 7：新信息前先给背景**

在要求读者考虑新信息之前先提供背景。这适用于所有层级——句子、段落、章节。

**弱**：「公式 3 表明当学习率满足……时收敛得到保证」
**强**：「为保证收敛，学习率必须满足公式 3 中的条件……」

#### 总结表

| 原则 | 规则 | 助记符 |
|-----------|------|----------|
| 主语动词临近 | 保持主语和动词靠近 | 「不要打断自己」 |
| 强调位置 | 句子末尾强调 | 「把最好的留到最后」 |
| 主题位置 | 句子开头给背景 | 「先说先说」 |
| 旧前新后 | 熟悉 → 陌生 | 「在已知基础上建立」 |
| 一单元一功能 | 每段一个观点 | 「一个容器一个想法」 |
| 动作在动词 | 用动词，不用名词化 | 「动词做事，名词坐着」 |
| 背景在新前 | 展示前先解释 | 「先搭台再唱戏」 |

## 微观层面写作技巧 | Micro-Level Writing Tips

### 来自 Ethan Perez (Anthropic)

这些实用的微观层面技巧可以提高句子和单词层面的清晰度。

#### 代词管理

**尽量少用代词**（「这个」「它」「这些」「那个」）。如果必须使用代词，用作形容词的修饰语加名词：

**弱**：「这表明模型收敛。」
**强**：「这个结果表明模型收敛。」

**弱**：「它提升了性能。」
**强**：「这个修改提升了性能。」

#### 动词位置

**把动词放在句子前面**以便更好地解析：

**弱**：「梯度在计算和归一化之后更新权重。」
**强**：「梯度在计算和归一化之后更新权重。」

#### 所有格展开

将所有格结构展开以提高清晰度：

**原文**：「X 的 Y」→ **展开**：「X 的 Y」

**前**：「模型在测试集上的准确率」
**后**：「模型在测试集上的准确率」

这并不总是更好，但当句子感觉不自然时，尝试展开。

#### 应删除的词汇

在几乎所有情况下删除这些填充词：
- 「实际上」
- 「有点」
- 「幸运地」/「不幸地」
- 「非常」/「真的」
- 「相当」
- 「基本上」
- 「本质上」
- 过多的连接词（「然而」「此外」「进一步」在不需要时）

#### 句子构造规则

1. **每句一个想法**——如果难以在一句话中表达一个想法，需要两句
2. **避免重复发音**——避免同一句子中出现发音相似的词
3. **每句话都要增加信息**——删除只是重复的句子
4. **始终使用主动语态**——指明行动者（「我们发现……」而不是「被发现……」）
5. **展开缩写**——为正式起见，「don't」→「do not」

#### 段落结构

- **第一句**：清晰陈述观点
- **中间句子**：用证据支持
- **最后一句**：强化或过渡

不要把关键信息埋在段落中间。

## 用词与精确性 | Word Choice and Precision

### 来自 Zachary Lipton

**消除模糊表达**，除非存在真正的不确定性：
- 删除「可能」和「可以」，除非必要
- 「提供*非常*紧的近似」充满不安全感
- 「提供*非常*紧的近似」充满不安全感
- 「提供紧密的近似」充满自信

**避免空洞的强调词**：
- 删除：非常、极其、高度、显著（除非是统计意义上的）
- 这些词表示不安全感，而不是力量。

### 来自 Jacob Steinhardt

**精确胜于简洁**：用具体的词替换模糊的词。

| 模糊 | 具体 |
|-------|----------|
| 性能 | 准确率、延迟、吞吐量 |
| 提升 | 准确率提升 X%、延迟降低 Y |
| 大型 | 10 亿参数、1 亿标记 |
| 快速 | 快 3 倍、50 毫秒延迟 |
| 良好结果 | 92% 准确率、0.85 F1 |

**一致的术语**：用不同的术语指代同一概念会造成混淆。

**选择一个并坚持使用**：
- 「模型」vs「网络」vs「架构」
- 「训练」vs「学习」vs「优化」
- 「样本」vs「例子」vs「实例」

### 词汇信号

**避免表示渐进式工作的词**：
- 永远不要：「结合」「修改」「扩展」「延伸」
- 改为：「开发」「提出」「引入」

**为什么**：「我们结合 X 和 Y」听起来像是把两个现有想法钉在一起。「我们开发了一种利用 X 进行 Y 的方法」听起来像是真正的贡献。

## 数学写作 | Mathematical Writing

### 来自 Ethan Perez

**为清晰起见展开所有格**：
- 弱：「X 的 Y」
- 强：「Y of X」

示例：「模型的准确率」→「准确率 of the model」

### 一般原则

1. **在定理前正式陈述所有假设**
2. **在证明旁提供直观解释**
3. **在整个论文中使用一致的符号**
4. **首次使用时定义符号**

### Notation Conventions

```latex
% Scalars: lowercase italic
$x$, $y$, $\alpha$, $\beta$

% Vectors: lowercase bold
$\mathbf{x}$, $\mathbf{v}$

% Matrices: uppercase bold
$\mathbf{W}$, $\mathbf{X}$

% Sets: uppercase calligraphic
$\mathcal{X}$, $\mathcal{D}$

% Functions: roman for named functions
$\mathrm{softmax}$, $\mathrm{ReLU}$
```

## 图表设计 | Figure Design

### 来自 Neel Nanda

图表应该讲述一个连贯的故事，即使读者跳过文字。许多读者最初确实会跳过文字。

### 设计原则

1. **图 1 至关重要**：通常是读者在摘要之后查看的第一个内容
2. **自包含的图注**：读者应该能够在不看正文的情况下理解图表
3. **图表内不要有标题**：图注起到这个作用（ICML/NeurIPS 规则）
4. **矢量图形**：图表使用 PDF/EPS，仅照片使用 PNG（600 DPI）

### 无障碍要求

8% 的男性有色觉障碍。你的图表必须对他们可用。

**解决方案**：
- 使用色盲友好的调色板：Okabe-Ito 或 Paul Tol
- 避免红绿组合
- 验证图表在灰度模式下可用
- 除了颜色之外，使用不同的线型（实线、虚线、点线）

### 工具

```python
# SciencePlots: 出版级样式
import matplotlib.pyplot as plt
plt.style.use(['science', 'ieee'])

# 或 Nature 风格
plt.style.use(['science', 'nature'])
```

## 常见错误与避免 | Common Mistakes to Avoid

### 结构错误

| 错误 | 解决方案 |
|---------|----------|
| 引言过长（>1.5 页）| 将背景移到相关工作 |
| 方法埋没（第 3 页之后）| 前置贡献，精简引言 |
| 缺少贡献要点 | 添加 2-4 个具体、可证伪的主张 |
| 实验没有明确主张 | 说明每个实验验证什么 |

### 写作错误

| 错误 | 解决方案 |
|---------|----------|
| 通用摘要开头 | 从你的具体贡献开始 |
| 术语不一致 | 每个概念选择一个术语 |
| 过度使用被动语态 | 使用主动语态：「我们展示」而不是「被展示」 |
| 到处模糊表达 | 除非真正不确定，否则要自信 |

### 图表错误

| 错误 | 解决方案 |
|---------|----------|
| 图表使用光栅图形 | 使用矢量（PDF/EPS） |
| 红绿配色方案 | 使用色盲友好调色板 |
| 图表内有标题 | 把标题放在图注中 |
| 图注需要正文 | 使图注自包含 |

### 引用错误

| 错误 | 解决方案 |
|---------|----------|
| 逐篇相关工作 | 按方法论组织 |
| 缺少相关引用 | 审稿人写了论文——慷慨引用 |
| AI 生成的引用 | 始终通过 API 验证 |
| 引用格式不一致 | 使用一致的 BibLaTeX 键 |

## 提交前检查清单 | Pre-Submission Checklist

提交前验证：

**叙事**：
- [ ] 能用一句话表达贡献
- [ ] 引言中三支柱（是什么/为什么/那又怎样）清晰
- [ ] 每个实验都支持一个具体主张

**结构**：
- [ ] 摘要遵循 5 句公式
- [ ] 引言 ≤1.5 页
- [ ] 方法从第 2-3 页开始
- [ ] 包含 2-4 个贡献要点
- [ ] 包含局限性部分

**写作**：
- [ ] 全文术语一致
- [ ] 没有通用的开头句子
- [ ] 删除不必要的模糊表达
- [ ] 所有图表都有自包含的图注

**技术**：
- [ ] 所有引用通过 API 验证
- [ ] 方法包含误差线
- [ ] 记录计算资源
- [ ] 说明代码/数据可用性
